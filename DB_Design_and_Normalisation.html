01

Key Database Vocabulary

These definitions must be precise — vague answers lose marks. Learn the exact wording.

TermDefinition
EntityA thing or object about which data is stored. Examples: Student, Book, Course, Teacher
AttributeA property or characteristic of an entity. Example: StudentID, Name, DateOfBirth
Relation / TableA two-dimensional structure of rows (tuples) and columns (attributes). All values in one column are of the same type.
Tuple / RecordA single row in a relation — one instance of an entity.
Primary Key (PK)An attribute, or minimal set of attributes, whose value uniquely identifies each tuple in a relation. Cannot be null.
Composite Primary KeyA PK made from two or more attributes. Neither attribute alone is sufficient — only the combination is unique.
Foreign Key (FK)An attribute in one relation whose values match the primary key values in another relation — creates a link between tables.
Referential IntegrityEvery FK value must correspond to an existing PK value in the referenced table. No orphaned records allowed.
Entity IdentifierAQA's term for primary key — the attribute(s) used to uniquely identify a tuple.
Candidate KeyAny attribute (or set) that could be used as a primary key — it uniquely identifies each row.
📝 Notation — Entity Descriptions
Student(StudentID, FirstName, LastName, DOB, *CourseID)
// Underlined = Primary Key    * prefix = Foreign Key

In AQA exams you must underline the PK and mark FKs with an asterisk (*).

02

Entity-Relationship Diagrams

An ERD is a visual model showing the entities in a system and how they relate. In AQA exams, entities are rectangles connected by lines labelled with the degree (cardinality) at each end.

Component

Entity

A rectangle labelled with the entity name. Represents a "thing" we store data about.

Component

Relationship

A line connecting two entities. Always label the degree at both ends.

Component

Degree / Cardinality

Shows how many instances can be involved: 1 (one side) or M (many side).

Component

Attributes

Described in entity notation format. Underline PK. Use * for FK.

Example ERD — Simple Library (1:M)

Library ERD — Member(1) loans many(M) Books, resolved via a Loan link table

Member(MemberID, MemberName, Email)
Loan(LoanID, *MemberID, *ISBN, DateOut, DueDate)
Book(ISBN, Title, Author, Genre)
03

Relationship Types & Degree

One-to-One (1:1)

One-to-Many (1:M) — Most Common ⭐

Many-to-Many (M:M) — Must Be Resolved ⚠️

Resolving M:M — The Link Table

Create a link table (junction / bridge / associative table) containing both PKs as foreign keys. Together they form a composite primary key.

✅ Key Insight

Both columns in the link table's composite PK are simultaneously foreign keys — they reference the PKs of both related tables. This is the standard pattern for resolving any M:M relationship.

04

Why Normalise?

An unnormalised database (flat file) storing all data in one table creates serious problems:

🔁

Update Anomaly

Changing a teacher's email means updating every row they appear in. Miss one → data becomes inconsistent.

➕

Insertion Anomaly

Cannot add a new course without a student enrolling first — every column is needed to create any row.

❌

Deletion Anomaly

Delete the last student on a course → all course data is permanently lost from the database.

♻️

Data Redundancy

Same data stored many times (e.g. teacher name in every row). Wastes space; increases inconsistency risk.

05

First Normal Form (1NF)

Rules for 1NF
  • All attribute values must be atomic (indivisible — one value per cell)
  • No repeating groups of columns
  • A primary key must be defined

The Thinking Process: UNF → 1NF

❓ Ask: Can a single attribute uniquely identify every row? No — StudentID S01 appears twice!
✅ Solution: Use (StudentID, CourseID) together as a composite primary key. One student can take many courses, so both are needed to pinpoint a row.

Enrolment(StudentID, CourseID, StudentName, DOB, CourseName, TeacherID, TeacherName, Grade)
// PK = (StudentID + CourseID) — composite
⚠ Still Not Fixed — Partial Dependencies Remain
  • StudentName, DOB → depend only on StudentID (not on CourseID)
  • CourseName, TeacherID, TeacherName → depend only on CourseID (not on StudentID)
  • Grade → depends on both StudentID AND CourseID ✅

These are called partial dependencies — fixed by 2NF.

06

Second Normal Form (2NF)

Rules for 2NF

Already in 1NF AND every non-key attribute is fully functionally dependent on the whole primary key — no partial dependencies.

Critical Terminology — Students Often Struggle Here

Concept

Functional Dependency

Attribute B is functionally dependent on A if knowing A always tells you B.

StudentID → StudentName

Written as: A → B

Problem (2NF violates)

Partial Dependency ⚠️

A non-key attribute depends on only PART of the composite PK — not all of it.

StudentID → StudentName (partial!)

CourseID is not needed to know StudentName

Target (2NF requires)

Full Functional Dependency ✅

A non-key attribute depends on the entire composite PK — removing any part breaks it.

StudentID + CourseID → Grade

Both parts of the PK needed for Grade

💡 Memory Trick for 2NF

"Does this attribute need ALL parts of the PK to identify it, or just PART of it?"

If it only needs PART → Partial Dependency → move it to its own table!

⚡ Important: 2NF only applies when there is a composite primary key. A single-attribute PK → already in 2NF if it was in 1NF.

Applying 2NF — Dependency Analysis

Non-key Attribute(s)Depends on...TypeAction
StudentName, DOB StudentID only ⚠ PARTIAL Move → Student table
CourseName CourseID only ⚠ PARTIAL Move → Course table
TeacherID, TeacherName CourseID only ⚠ PARTIAL Move → Course table
Grade StudentID + CourseID (both) ✅ FULL Keep in Enrolment

Result: 2NF — Three Tables

Student(StudentID, StudentName, DOB)
Course(CourseID, CourseName, TeacherID, TeacherName) // ← TeacherName still a problem!
Enrolment(StudentID, CourseID, Grade) // composite PK
07

Third Normal Form (3NF)

Rules for 3NF

Already in 2NF AND there are no transitive dependencies — every non-key attribute must depend only and directly on the primary key.

Transitive Dependency

Definition

A non-key attribute (C) depends on another non-key attribute (B), which in turn depends on the PK (A).

CourseID (PK) → TeacherID → TeacherName

TeacherName depends on TeacherID (a non-key attr), not directly on CourseID (the PK). This is the transitive dependency.

Fix: Move TeacherName into its own Teacher table with TeacherID as PK. Course keeps TeacherID as a FK.

Result: 3NF — Four Tables

Student(StudentID, StudentName, DOB)
Enrolment(StudentID, CourseID, Grade) // composite PK — both are also FKs
Course(CourseID, CourseName, *TeacherID)
Teacher(TeacherID, TeacherName)

Final 3NF Entity-Relationship Diagram

Final 3NF database — four tables, all anomalies eliminated

✅ 3NF Achieved — What Changed?
  • TeacherName moved into a new Teacher table — now depends only on TeacherID (its own PK)
  • Course table keeps *TeacherID as a Foreign Key linking to Teacher
  • Every non-key attribute in every table now depends only on that table's primary key

UNF
Unnormalised Raw data in one table. No PK. Repeating groups. All anomalies present.
1NF
First Normal Form Atomic values. PK defined (composite if needed). No repeating groups.
Removes: repeating groups
2NF

Removes: partial dependencies
3NF
Third Normal Form 2NF + no transitive dependencies. Non-key attrs depend ONLY on PK.
Removes: transitive dependencies

"Every non-key attribute must depend on THE KEY, THE WHOLE KEY, and NOTHING BUT THE KEY."

Exam Tips

📝

Define Terms Precisely

"A primary key is an attribute (or set of attributes) that uniquely identifies each tuple in a relation." Include key vocabulary — vague answers score zero.

🔗

Label Both Ends of ERD Lines

Always write 1 or M at both ends of every relationship line. Forgetting either end loses the mark.

⚠️

2NF Only With Composite PK

If the PK is a single attribute, there can be no partial dependencies — the relation is automatically in 2NF. Only check 2NF when PK has 2+ attributes.

🔄

Partial vs Transitive

Partial = non-key depends on part of composite PK → 2NF issue.
Transitive = non-key depends on another non-key → 3NF issue.

✅

Underline PKs, Mark FKs

In entity description notation, always underline the primary key attribute(s) and prefix foreign keys with *. Easy marks often lost here.

🧠

Resolve All M:M Relationships

In a normalised design question, an unresolved many-to-many is always wrong. Always create the link table with a composite PK.

The Designing Process — Quick Reminder

STEP 1

Identify Entities

Nouns in the scenario → entities (rectangles)

STEP 2

Find Attributes

What data is stored? Pick/invent a PK for each

STEP 3

Find Relationships

Check every pair: is it 1:1, 1:M, or M:M?

STEP 4

Resolve M:M

Create link tables with composite PKs

STEP 5

Add FKs

FK goes on the many-side of 1:M relationships
