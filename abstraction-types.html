Key principle: Abstraction always involves removing or hiding detail. The question is what kind of detail is being hidden, and why. Understanding the differences between the types below is essential for Paper 2.

1
Representational Abstraction
4.4.1.3
Definition

Removing unnecessary details from a representation of a real-world object or problem, keeping only the information relevant to solving the problem at hand.

Classic Example

The London Underground Tube map — real geography and distances are distorted or ignored. Only what a passenger needs is kept: station order, line connections, and interchanges.

Further Examples
  • Weather model uses averaged grid values, not individual air molecules
  • Student class omits hair colour — irrelevant to the problem
Exam questions may ask you to identify what has been removed in a given abstraction, or why a particular detail was excluded. Always link to the purpose of the model.
2
Abstraction by Generalisation / Categorisation
4.4.1.3
Definition

Grouping objects or concepts by their common characteristics to form a hierarchy of 'is a kind of' relationships.

Example

Vehicle superclass contains Car, Lorry, Motorcycle — all share engine, wheels, speed. Common code (e.g. calculateFuelCost) written once and inherited by all subclasses.

Further Examples
  • Animal generalises Dog, Cat, Bird — all are a kind of Animal
  • Shape generalises Circle, Rectangle, Triangle
  • Person entity generalises Student and Teacher in a DB
Closely linked to inheritance in OOP. Think about what all members of a group have in common — that forms the basis for a parent class.
3
Information Hiding
4.4.1.4
Definition

Concealing all details of an object or system that do not contribute to its essential characteristics. Users only see the interface, not the internal implementation.

Example

A Stack exposes push(), pop(), peek(). The internal implementation (array or linked list, top pointer) is hidden. The programmer only uses the public operations.

Further Examples
  • Car driver uses steering wheel / pedals, not the combustion engine
  • sort() hides whether it uses quicksort or mergesort
  • Private instance variables accessed only via public getters/setters
Information hiding is the PRINCIPLE of not exposing internal details. Encapsulation is the TECHNIQUE (access modifiers) used to enforce this in OOP. Be precise about which is which.
4
Procedural Abstraction
4.4.1.5
Definition

Abstracting away the actual values used in a computation to produce a general computational method — a procedure. Actual values are supplied as parameters when called.

Example

calculateArea(width, height) — the computation method (multiply width × height) is abstracted once. Specific values (5 and 3) are supplied as arguments at call time.

Further Examples
  • drawCircle(x, y, radius) — method captured once, coordinates supplied per call
  • printHeading(title) — pattern of printing a heading, any text passed in
  • swap(a, b) — pattern captured without committing to specific values
Procedural abstraction produces a procedure (may not return a value). The computation method IS visible in the code. Contrast with functional abstraction which also hides the method.
5
Functional Abstraction
4.4.1.6
Definition

Takes procedural abstraction further by also hiding the particular computation method used. The function is defined purely by its input-output relationship — a black box.

Example

square_root(n) — caller provides a number and receives the result. Whether Newton's method, binary search, or a lookup table is used internally is completely hidden.

Further Examples
  • Built-in sort() — you pass a list, receive a sorted list; algorithm hidden
  • DB query — pass search criteria, receive records; index structures hidden
  • sin(x) in a library — approximation method entirely concealed
Functional abstraction hides BOTH the specific values AND the method of computation. The function is a black box — given this input, you will always get this output.
6
Data Abstraction
4.4.1.7
Definition

Isolating how a compound data object is used from the details of how it is constructed and stored. Users interact through a defined interface — the ADT.

Example

Stack ADT — constructed from array + integer (top pointer), but user only calls push, pop, isEmpty. Could be reimplemented with a linked list; abstraction stays the same.

ADTs to Know
  • Queue: enqueue, dequeue, isEmpty → circular array implementation
  • Dictionary: get(key), put(key,val), delete(key) → hash table or BST
  • Graph: addVertex, addEdge, getNeighbours → adjacency matrix or list
For each ADT, know what it does (abstract operations) AND one way it could be concretely implemented.
7
Problem Abstraction / Reduction
4.4.1.8
Definition

Removing details from a problem until it is represented in a way that can be solved — because it reduces to a problem that has already been solved.

Example

The Travelling Salesman Problem reduces to finding the shortest Hamiltonian cycle in a weighted graph — a known graph theory problem — by stripping away real-world geographic detail.

Further Examples
  • Lecture scheduling → graph colouring problem
  • Route-finding (satnav) → shortest path in weighted graph (Dijkstra)
  • Inventory management → bin-packing problem
Focus on what is removed and how the resulting problem matches a known solvable form. Computer scientists recognise when a new problem is an instance of a solved one.
8
Decomposition
4.4.1.9
Definition

Breaking a large, complex problem down into a number of smaller sub-problems, each accomplishing an identifiable task, continuing until each is simple enough to implement directly.

Example

School management system → manage students / staff / timetable / reports → manage students → add student / remove student / update record / search. Each leaf task is a self-contained module.

Further Examples
  • Compiler → lexical analysis, parsing, semantic analysis, code generation
  • Game → get input, update state, render graphics, play sound, check win
  • Payroll → gross pay, deductions, net pay, payslip, update records
Decomposition is a design technique often tested alongside structure charts or top-down design. Breaking a problem into sub-problems; composition is the inverse.
9
Composition
4.4.1.10
Definition

The inverse of decomposition — combining simpler procedures or data objects to build more complex compound procedures or data structures.

Example

makeCoffee() composed from boilWater(), grindBeans(), brewCoffee(), addMilk(). A binary tree is composed of node objects, each composed of a data value and two child references.

Further Examples
  • LinkedList composed of Node objects (data + next reference)
  • Quicksort composed of a partition procedure + recursive calls
  • drawScene() composed of drawBackground(), drawCharacters(), drawHUD()
In OOP design, 'favour composition over inheritance' means building a class that contains instances of other classes (has-a) rather than inheriting from them (is-a).
10
Automation
4.4.1.11
Definition

Putting abstract models into action to solve problems. Requires: creating algorithms; implementing them as code; implementing models as data structures; and executing the code.

Example

Weather forecasting: the atmosphere is modelled as temperature/pressure grids → numerical algorithms solve differential equations → implemented as code → executed on supercomputers → produces forecasts.

Four Stages
  • Model (abstraction of the real world)
  • Algorithm (acts on the model)
  • Implementation (code + data structures)
  • Execution (produces the solution)
Automation questions require you to explain ALL FOUR stages. A common error is to describe only the code — remember automation begins with the abstract model, not the program.

Key Distinctions — Frequently Confused in Exams

Information Hiding vs Encapsulation

Information hiding is the principle — internal details should not be exposed.

Encapsulation is the mechanism used in OOP to enforce this principle, using access modifiers (private, public, protected).

You can have information hiding without OOP (e.g. a module that exposes only certain functions), but in OOP, encapsulation is the standard way to implement it.

Procedural vs Functional Abstraction

Procedural abstraction removes specific values to produce a general procedure — the computation method is still visible in the code.

Functional abstraction goes further: the computation method is also hidden. The user only knows the input-output behaviour, not how the output is produced. Think of it as a black box.

Decomposition vs Composition

Decomposition breaks a complex problem down into simpler sub-problems (top-down design).

Composition builds complex systems up from simpler components (bottom-up construction).

In practice, software design often uses both: decompose the problem first to understand its structure, then compose solutions from reusable modules.

Data Abstraction vs Information Hiding

Data abstraction focuses specifically on separating how a data structure is used (abstract operations) from how it is implemented (concrete representation).

Information hiding is a broader principle that applies to any component — whether data or procedural.

An ADT achieves data abstraction; it also uses information hiding to conceal its implementation.

Practice Questions — AQA Exam Style

[2 marks]
A programmer creates a function called calculateVAT(price) that returns the VAT-inclusive price. Identify the type of abstraction being used and explain your answer.
Procedural abstraction (1). The specific values (price) are abstracted away; the general computation method (multiply by VAT rate) is captured as a named procedure that accepts the price as a parameter (1).
[4 marks]
A Stack ADT is implemented internally using an array and an integer variable. A programmer uses the push() and pop() methods without knowing the internal implementation. State two types of abstraction demonstrated in this scenario and explain each.
Data abstraction: the internal implementation (array + pointer) is separated from the abstract operations (push/pop) used by the programmer (2). Information hiding: the internal details of the stack are concealed behind a defined interface; the programmer uses the stack without knowledge of its implementation (2).
[3 marks]
A software team is designing a hospital management system. They identify the following subsystems: patient records, appointment booking, staff rota, and billing. State the type of abstraction being used and explain how it helps with the design process.
Decomposition (1). The complex hospital system is broken into identifiable sub-problems (1). Each sub-problem can be designed, developed and tested independently, reducing complexity and enabling parallel development (1).
[4 marks]
Explain the difference between representational abstraction and problem abstraction/reduction, using an example of each.
Representational abstraction removes unnecessary detail from a model, e.g. a tube map removes real geography but keeps station order (2). Problem abstraction strips domain detail until the problem becomes one that has already been solved, e.g. a scheduling problem reduced to graph colouring (2).
[4 marks]
Describe what is meant by automation in the context of computer science. Your answer should refer to all four stages of the automation process.
Automation requires: creating an abstract model of the real-world problem (1); designing algorithms to act on the model (1); implementing the algorithms and model as code and data structures (1); executing the code to produce a solution (1).
[3 marks]
A programmer writes a procedure swap(a, b) that exchanges the values of two variables. A second programmer writes a function max(a, b) that returns the larger of two numbers, hiding the comparison method used. Distinguish between the types of abstraction demonstrated in each case.
swap(a, b) is procedural abstraction: specific values are removed and the general swap method is captured as a procedure (1). max(a, b) is functional abstraction: not only are the specific values abstracted away, but the method of comparison is also hidden — the caller only sees the input-output relationship (2).
