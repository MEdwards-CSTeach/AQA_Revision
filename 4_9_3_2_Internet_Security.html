🔒 Section 1 — Firewalls

4.9.3.2

A firewall monitors and controls incoming and outgoing network traffic based on predetermined rules.

Packet Filtering

Examines packet headers. Blocks/allows based on rules about:

  • Source/destination IP address
  • Port number
  • Protocol type

💡 Rules can block specific ports (e.g. block port 25 to prevent spam)

Stateful Inspection

Tracks the state of active connections. Only allows packets that belong to an established, legitimate connection.

💡 More secure than packet filtering alone — blocks unsolicited incoming packets

Proxy Server

Acts as an intermediary. Internal computers never communicate directly with the internet — all requests go through the proxy first.

💡 Hides internal IP addresses; can cache and filter content

⚠️ Exam tip "Explain 4 ways a firewall can protect a LAN" — give a different mechanism each time; don't repeat firewall types for more than 1 mark each.

🔐 Section 2 — Encryption

4.9.3.2

Symmetric Encryption

Same key used to encrypt and decrypt

  • ✅ Fast — suited to large data
  • ✅ Simple to implement
  • ❌ Key distribution problem: How do you securely share the key in the first place?

Example: AES

Asymmetric Encryption

Different key pair (public + private)

  • Public key: shared openly
  • Private key: kept secret
  • Encrypt with receiver's PUBLIC key
  • Decrypt with receiver's PRIVATE key
  • ❌ Slow — not suited for large data
  • ✅ Solves key distribution problem

Example: RSA

🔑 Key Exchange in Practice (HTTPS)

4.9.3.2

Both encryption methods are combined — asymmetric is used to securely share a symmetric key:

  1. 1
    Server sends its public key to the clientThe public key is included in the server's digital certificate
  2. 2
    Client generates a random symmetric session keyThis will be used for the bulk of the encrypted communication
  3. 3
    Client encrypts the session key using the server's PUBLIC keyOnly the server can decrypt this — it's the only one with the matching private key
  4. 4
    Encrypted session key sent to the serverEven if intercepted, it cannot be read without the server's private key
  5. 5
    Server decrypts using its PRIVATE keyNow both parties know the symmetric session key
  6. 6
    Both now use the symmetric session key for fast encryptionAll subsequent communication is fast symmetric encryption
⚠️ Exam tip Asymmetric encryption is SLOW — it's used only to share the session key, not to encrypt the whole message.

📜 Section 3 — Digital Certificates

4.9.3.2

A digital certificate proves that a public key really does belong to the claimed entity.

Certificate Contains

  • Owner's public key
  • Owner's name / domain
  • Issuing Certificate Authority (CA)
  • Expiry date
  • Digital signature of the CA
  • Serial number

Obtaining a Certificate

  1. Entity generates a public/private key pair
  2. Sends public key + identity details to a Certificate Authority (CA)
  3. CA verifies identity, then signs and issues the certificate

Using / Verifying a Certificate

  1. Client receives server's certificate
  2. Client checks CA's digital signature (CA's public key is built into the browser)
  3. If valid → client trusts the public key → encrypted communication begins (HTTPS)
💡 Exam tip The CA's public key is pre-installed in browsers — this is how they verify the certificate without contacting the CA each time.

✍️ Digital Signatures — How They Work

4.9.3.2

Purpose: Prove the message came from the claimed sender and was not tampered with.

📤 SENDER (A) — Signing

  1. Hash the message — creates a message digest
  2. Encrypt the hash using sender's PRIVATE key → This is the digital signature
  3. Send message + signature (message encrypted with receiver's PUBLIC key)

📥 RECEIVER (B) — Verifying

  1. Decrypt message using receiver's PRIVATE key
  2. Decrypt the signature using sender's PUBLIC key → Gets original hash
  3. Hash the received message independently
  4. Compare hashes — If they match → authentic & unaltered
⚠️ Key distinction The sender signs with their PRIVATE key. The receiver verifies with the sender's PUBLIC key. A hash is one-way — it cannot be reversed. Encryption CAN be reversed with the correct key.

🐛 Section 4 — Malware

4.9.3.2

🦠 Virus

How it works: Attaches itself to a legitimate program or file. Spreads when the infected file is shared.

Vulnerability exploited: Users opening infected email attachments or running infected files.

Requires human action to spread

🪱 Worm

How it works: Self-replicating — spreads automatically across networks without needing to attach to a file.

Vulnerability exploited: Network vulnerabilities and software bugs. Can cause network congestion.

Spreads without human action

🐴 Trojan

How it works: Disguises itself as legitimate software. Does not self-replicate.

Vulnerability exploited: Users' trust — tricks users into installing it voluntarily (social engineering).

Does not self-replicate

🛡️ Section 5 — Countermeasures

4.9.3.2

🧪 Code Quality

  • Prevent buffer overflows — validate input so programs cannot write to unallocated memory
  • Code reviews — independent checking by another programmer
  • Security testing — test for vulnerabilities before release
  • Use code analysis software to detect flaws

👁️ Monitoring

  • Firewall — block packets from high-risk sources; stateful inspection
  • Proxy server — checks downloaded files before they reach users
  • Spam filters — block suspicious emails and attachments
  • Web filters — check sites against known malware lists
  • Digital certificates — verify source of downloads

🔒 Protection

  • Auto-update OS and apps to patch vulnerabilities
  • Use a sandbox/VM to safely execute unknown programs
  • Set access rights to minimise damage from malware
  • Disable macros in documents from external sources
  • Encrypt sensitive files; take offline backups
⚠️ Exam tip The question will often specify monitoring/protection/code quality — answer within THAT category and always describe HOW, not just name the measure.

⚠️ Top Exam Tips

📌 Always describe HOW

Naming "firewall" or "antivirus" without explaining how it works earns zero. Always say what it DOES.

📌 Symmetric ≠ Asymmetric

Symmetric = same key both ends. Asymmetric = public key encrypts, private key decrypts. Don't mix them up.

📌 Digital Signature = private key

The sender signs with their PRIVATE key. The receiver verifies with the sender's PUBLIC key.

📌 Hashing ≠ Encryption

A hash is one-way — it cannot be reversed. Encryption CAN be reversed with the correct key.

📌 Don't repeat marks

If asked for 4 different firewall methods, give 4 different ones. Multiple points about "packet filtering" = 1 mark max.

📌 Read the category

If the question says "from the area of code quality" — stay in that category. Answering with monitoring measures won't score.
